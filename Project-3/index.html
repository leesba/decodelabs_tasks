Database Blueprint

Schema Design

The interns table — every row is a record, every column is a typed, constrained attribute.

⬡
interns
MySQL Table · decodelabs_db
PK id INT AUTO_INCREMENT · NOT NULL
col name VARCHAR(100) NOT NULL
UQ email VARCHAR(100) NOT NULL · UNIQUE
col batch VARCHAR(10) DEFAULT '2026'
col domain VARCHAR(100) NOT NULL
col status ENUM 'active' | 'inactive'
TS created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
🔑

Primary Key

Uniquely identifies every row. id is auto-incremented — never null, never repeated.

✦

UNIQUE Constraint

No two interns can share the same email — enforced at the database level, not the app.

⚙

ENUM Type

status only accepts 'active' or 'inactive'. Any other value is rejected by MySQL.

🚫

NOT NULL

Mandatory fields. Even if your app forgets to send them — MySQL refuses the row.

setup.sql DDL — Data Definition Language 
CREATE DATABASE IF NOT EXISTS decodelabs_db
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

USE decodelabs_db;

CREATE TABLE IF NOT EXISTS interns (
  id         INT          AUTO_INCREMENT PRIMARY KEY,
  name       VARCHAR(100) NOT NULL,
  email      VARCHAR(100) NOT NULL UNIQUE,
  batch      VARCHAR(10)  NOT NULL DEFAULT '2026',
  domain     VARCHAR(100) NOT NULL,
  status     ENUM('active', 'inactive') DEFAULT 'active',
  created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);

Core Database Theory

Key Concepts

SQL vs NoSQL

SQL
  • Structured tables with fixed schema
  • Rows & columns — like a spreadsheet
  • ACID transactions — data is always consistent
  • Strong relationships via Foreign Keys
  • MySQL · PostgreSQL · SQLite
Use when: Data is relational and structure matters
VS
NoSQL
  • Flexible / schema-less documents
  • JSON-like objects, key-value, graphs
  • Horizontal scalability — millions of records
  • Eventual consistency
  • MongoDB · Redis · Cassandra
Use when: Data is unstructured or scale is massive

Table Relationships

1 : 1
One-to-One

One intern has exactly one profile card. One profile belongs to exactly one intern.

intern.id → profile.intern_id
1 : ∞
One-to-Many

One batch has many interns. Each intern belongs to exactly one batch.

batch.id → intern.batch_id
∞ : ∞
Many-to-Many

Interns take many courses. Courses have many interns. Needs a junction table.

intern_courses(intern_id, course_id)
Data Protection

SQL Injection & Defense

The most common database attack — and how parameterized queries stop it cold.

VULNERABLE String Concatenation 
$email = "' OR 1=1 --";

// DANGER: User input injected directly into SQL
$query = "SELECT * FROM interns
  WHERE email = '" . $email . "'";

// Actual SQL that runs:
SELECT * FROM interns WHERE email = ''
OR 1=1 --'
-- Returns ALL rows! Data is exposed.
PROTECTED Parameterized Query 
$email = "' OR 1=1 --";

// SAFE: ? is a placeholder, not injected SQL
$stmt = $pdo->prepare(
  "SELECT * FROM interns WHERE email = ?"
);
$stmt->execute([$email]);

// Input is treated as DATA, never as SQL code
// No rows returned — attack neutralized ✓
⚡
The Parameterized Rule: Never concatenate user input into SQL strings. Always use prepare() + execute([]). The database engine treats ? as a data slot — not executable SQL. Attack neutralized at the engine level.
