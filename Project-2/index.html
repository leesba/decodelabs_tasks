Anatomy of Action

HTTP Methods

Resources are Nouns. Methods are Verbs.

GET

Retrieval

Fetch data from the server. Never modifies state. Safe to repeat.

Safe Idempotent No Body
GET /api/users
POST

Creation

Send data to create a new resource. Each call may create a new record.

Unsafe Non-idempotent Body
POST /api/users
PUT

Replacement

Replace an existing resource entirely. Same call twice = same result.

Unsafe Idempotent Body
PUT /api/users/1
DELETE

Removal

Remove a resource from the server. Once deleted, returns 404 on repeat.

Unsafe Idempotent No Body
DELETE /api/users/1

JSON Rules

  • { }
    Objects

    Key-value pairs wrapped in curly braces

  • [ ]
    Arrays

    Ordered lists wrapped in square brackets

  • " "
    Keys must be strings

    Always wrapped in double quotes

  • ≠,
    No trailing commas

    Last item in an object or array has no comma

Critical Vocabulary

HTTP Status Codes

Don't force the client to guess. Use the semantic code.

2xx · Success
200 OK Request successful
201 Created Resource created (POST)
204 No Content Success, no response body
4xx · Client Error
400 Bad Request Malformed input
401 Unauthorized Who are you?
403 Forbidden I know you, but no.
404 Not Found Resource missing
429 Too Many Requests Rate limit exceeded
5xx · Server Error
500 Internal Server Error Server broke. Your fault. Fix it.

The Blood-Brain Barrier of your API

01

Syntactic Validation

Is the format correct? Check structure before any processing begins.

  • Required fields present?
  • Correct data types?
  • Valid email / phone format?
02

Semantic Validation

Is the logic valid? Check meaning and business rules after syntax passes.

  • Does user exist in DB?
  • Is the date in the future?
  • Does the role allow this action?
03

Authentication & Authorization

AuthN = Who are you? AuthZ = What are you allowed to do?

  • Valid token or session?
  • Correct permissions scope?
  • Rate limit not exceeded?
